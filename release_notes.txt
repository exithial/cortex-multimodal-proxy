# Cortex Multimodal Proxy v3.3.0 — Release Notes

## What's new

Cortex Multimodal Proxy v3.3.0 adds an informational dashboard. It captures every request that flows through the proxy — what model, how many tokens in and out, how long it took, whether it was served from cache, whether it errored — and surfaces the totals and the recent activity on a single page you can open in your browser.

If you run the proxy and want to know what is actually costing you money, what is hitting cache, and what is failing, the dashboard answers those questions at a glance.

Open `http://<your-proxy-host>:7777/dashboard/` once the proxy is running and `DASHBOARD_ENABLED=true` is set in `.env`.

The dashboard shows:

- Six hero cards — total tokens (split in/out), USD cost, request count (ok / errors), cache hit ratio (with hits / misses), error rate, uptime.
- A 24-hour or 30-day time-series chart of tokens in and out, toggleable with two buttons.
- A per-model breakdown table sorted by request count — model name, brain, in/out tokens, USD, requests, errors, cache hits, p50 latency, p95 latency.
- A live log tail showing the last 200 lines of `combined.log` and `error.log`, with a level filter (info+, warn+, error, debug) and a substring search box.
- A footer with version, mode, providers, and how many log lines the tail is reading.

The dashboard polls the proxy every 10 seconds, so the numbers you see are always close to live. It uses the same single port and the same Express app as the rest of the proxy, so there is no new process to manage and no new auth surface beyond what the proxy already has.

## Why this matters

- **Visibility** — until now, the proxy told you almost nothing about what it was doing. You had to scrape logs to answer "what did I spend on OpenCode Go this week?" The dashboard answers that question in one click.
- **Cache debugging** — the dashboard shows cache hits separately from misses, so you can tell at a glance whether the prompt-cache hints you set up are actually working.
- **Catch regressions early** — error rate, p95 latency, and per-model breakdown let you see when a particular provider starts misbehaving without having to wait for the user to complain.

## What's stable

- The proxy's public API (`/v1/chat/completions`, `/v1/messages`, `/v1/models`, `/health`, `/v1/cache/stats`) is unchanged.
- BRAIN_MODE behavior, model selection, multimodal pipeline, and retry policy are unchanged.
- If you set `DASHBOARD_ENABLED=false` (the new default), the dashboard is fully off — no DB writes, no extra endpoint, zero overhead.
- Existing clients do not need to change.

## Security

- The dashboard reads token counts, USD cost, and the last 200 lines of `combined.log` and `error.log`. Logs are server-side redacted before they reach the browser: Bearer tokens, `sk-...` API keys, email addresses, `data:image` base64 URIs, and long base64 blobs are all stripped.
- The dashboard is unauthenticated, like the rest of the proxy. **Keep it on a trusted network** (Tailscale, VPN, `127.0.0.1`). Never expose it on a public IP or a shared LAN. If you need it on a broader network, put a reverse proxy with auth in front (Caddy, nginx with `auth_basic`, Cloudflare Tunnel with Access, etc.).
- `DASHBOARD_ENABLED` now defaults to `false`. Operators must consciously enable the dashboard after reading this warning.

## Other improvements in this release

- `MINIMAX_TIMEOUT_MS=300000` (5 minutes) — long thinking runs on large contexts no longer hit an implicit zero-second axios timeout. If you don't set it, it falls back to `SENSES_TIMEOUT_MS`.
- `MINIMAX_INPUT_PRICE` / `MINIMAX_OUTPUT_PRICE` env vars — set your real per-token rates (USD per million tokens) so the dashboard reports actual cost for `MiniMax-M3` traffic. The dashboard starts a warning at boot if `MINIMAX_API_KEY` is set but both prices are zero.
- MiniMax-M3 streaming fix — every chunk in a stream now shares the same `chatcmpl-<uuid>` id instead of generating a fresh id per chunk. Clients like OpenCode use the chunk id to accumulate context token counts; the per-chunk-id bug was causing OpenCode's context counter to reset to zero on every response.

## Upgrading

1. Pull the new image / checkout the tag.
2. Set `DASHBOARD_ENABLED=true` in `.env` if you want the dashboard. Leave it `false` (the default) to disable.
3. Set `MINIMAX_INPUT_PRICE` and `MINIMAX_OUTPUT_PRICE` if you use MiniMax-M3 and want real cost tracking.
4. If your proxy is on Docker, add the new `proxy-data` named volume (already declared in `compose.yml`). The first start creates the SQLite database inside the volume.
5. Restart the container. No client reconfiguration required.
