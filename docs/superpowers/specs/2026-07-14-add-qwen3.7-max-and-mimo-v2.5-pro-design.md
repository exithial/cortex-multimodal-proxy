# Spec: Add `proxy/qwen3.7-max` and `proxy/mimo-v2.5-pro` Brains

**Date**: 2026-07-14
**Status**: Draft (pending user review)
**Scope**: Extend the Cortex Multimodal Proxy brain registry with two new text-only models served via OpenCode Go, gaining image vision through the existing MiMo V2.5 senses layer.

---

## Goal

Expand `BRAIN_MODELS` in `src/services/brainRegistry.ts` to support two additional text-only brains from OpenCode Go so that OpenCode and Claude Code clients can route requests to them and still get image vision via the existing MiMo V2.5 senses layer.

### Success Criteria

1. `proxy/qwen3.7-max` and `proxy/mimo-v2.5-pro` are routable through `/v1/chat/completions` and `/v1/messages` like the existing brains.
2. Image-bearing requests to either new brain trigger `mimoSensesService.describeImage()` (no Gemini for images) — same path as the existing 2 brains.
3. Audio/video/PDF requests still fall back to `geminiService.analyzeContent()` for both new brains.
4. Streaming works for both new brains on both OpenAI and Anthropic-format responses.
5. No breaking changes to existing model routes (`proxy/glm-5.2`, `proxy/deepseek-v4-pro`, `mimo-v2.5`).
6. No new alias added for Claude Code (`haiku`/`sonnet`/`opus` mappings stay as-is).
7. `proxy/mimo-v2.5-pro` coexists with the existing `mimo-v2.5` passthrough — no replacement.
8. `npm run build`, `npm run test:unit`, `npm run lint` all pass.
9. Documentation in `README.md`, `CLAUDE.md`, `MODELS.md` is updated to reflect the new brain list and combined pricing.

## Background

### Current State (v3.0.0)

The proxy implements "Cortex Sensorial v3":

- **2 text-only brains** in `BRAIN_MODELS`: `proxy/glm-5.2` and `proxy/deepseek-v4-pro`. Both use the OpenAI-format endpoint and 800K context.
- **1 passthrough**: `mimo-v2.5` (natively multimodal, no proxy prefix, hits OpenAI-format endpoint directly).
- **MiMo V2.5 senses** (`mimoSensesService`): describes images for any `proxy/*` brain (`multimodalProcessor.ts:42-46`).
- **Gemini fallback** (`geminiService`): handles audio/video/PDF when needed.
- The endpoint discriminator `endpoint: "openai" | "anthropic"` is already present in `BrainModelEntry` and the conversion code (`openAIToAnthropicPayload` at `src/services/opencodeGoService.ts:25-118`) is already implemented and tested for Anthropic-format brains (Minimax-M3 internally), even though no Anthropic-format brain is currently registered.

### Why Two New Brains

- **Qwen3.7 Max**: Qwen flagship, top of OpenCode Go's range per their official docs. Higher capability, higher price. Useful for tasks where brain strength > cost.
- **MiMo V2.5 Pro**: Xiaomi's "Pro" tier — a different SKU than the existing `mimo-v2.5` passthrough. The existing passthrough is natively multimodal and cheap ($0.14/$0.28). The "Pro" version is text-only and pricier ($1.74/$3.48), but smarter at code/reasoning. Adding it as a brain (not passthrough) lets clients get a stronger Xiaomi model with vision via the MiMo V2.5 senses layer.

## Design

### 1. Architecture (No Changes)

The data-flow diagram is unchanged. The new brains slot into the existing pipeline:

```
Client (OpenCode / Claude Code)
        ↓ /v1/chat/completions or /v1/messages
src/index.ts route handlers
        ↓
multimodalProcessor.processMultimodalContent
   ├─ passthrough (mimo-v2.5) → direct
   └─ proxy/* → if image: mimoSensesService.describeImage
                          if audio/video/PDF: geminiService.analyzeContent
        ↓
opencodeGoService.createChatCompletion (or .chatCompletionStream)
        ↓
[OpenCode Go] POST /v1/messages      (qwen3.7-max)
        OR
[OpenCode Go] POST /v1/chat/completions  (mimo-v2.5-pro)
```

**Key insight**: the multimodal senses path is keyed by `modelName.startsWith("proxy/")` (`multimodalProcessor.ts:45`), so vision is enabled automatically for any new `proxy/*` brain without touching the processor.

### 2. `BRAIN_MODELS` Additions

In `src/services/brainRegistry.ts`, append two entries:

```ts
"proxy/qwen3.7-max": {
  upstream: "qwen3.7-max",
  context: 1_048_576,
  maxOutput: 65_536,
  thinking: true,
  inputPrice: 2.50,
  outputPrice: 7.50,
  endpoint: "anthropic",
  multimodal: false,
},
"proxy/mimo-v2.5-pro": {
  upstream: "mimo-v2.5-pro",
  context: 1_048_576,
  maxOutput: 65_536,
  thinking: true,
  inputPrice: 1.74,
  outputPrice: 3.48,
  endpoint: "openai",
  multimodal: false,
},
```

Notes:
- `endpoint: "anthropic"` for Qwen — handled by existing `openAIToAnthropicPayload` in `opencodeGoService.ts:25-118`.
- `endpoint: "openai"` for MiMo V2.5 Pro — same path as the existing two brains.
- `thinking: true` — both are reasoning models; payload builder already emits `thinking: { type: "enabled" }`.
- `multimodal: false` — vision comes via MiMo senses, not natively.
- 1M context (vs. 800K for existing brains) requires no special handling; the `truncateMessages` helper in `messageTransforms.ts` already uses each brain's `context` field.
- 65K `maxOutput` chosen as a safe default since both OpenCode Go docs and llm-stats confirm ≥65K. Adjust if a hard cap surfaces in integration testing.

### 3. `PASSTHROUGH_MODELS` (No Changes)

`mimo-v2.5` stays the only passthrough. `mimo-v2.5-pro` is a brain, not a passthrough — by user decision (coexistence).

### 4. Coexistence with `mimo-v2.5` Passthrough

- `mimo-v2.5` (no prefix) → passthrough → OpenAI-format direct to upstream, natively multimodal, $0.14/$0.28 per 1M.
- `proxy/mimo-v2.5-pro` → brain → OpenAI-format direct to upstream, vision via MiMo senses, $1.74/$3.48 per 1M.

Both are listed in `/v1/models`. No collision: `isPassthrough()` and `getBrainEntry()` use exact-string match (`Object.hasOwn`/`Set.has`), and the prefix is structurally distinct.

### 5. Claude Code Aliases (No Changes)

- `haiku` → `mimo-v2.5` (passthrough)
- `sonnet` → `proxy/deepseek-v4-pro` (default)
- `opus` → `proxy/glm-5.2` (default)

No new aliases. Clients that want the new brains in Claude Code can set `CLAUDE_HAIKU_MODEL`, `CLAUDE_SONNET_MODEL`, or `CLAUDE_OPUS_MODEL` env vars to `proxy/qwen3.7-max` or `proxy/mimo-v2.5-pro` (documented in `README.md` env table).

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|-----------|
| Qwen3.7 Max upstream returns errors despite being listed | Low (was previously removed for unreliability per MODELS.md) | Integration test against real API; if it fails, keep entry but document known-flaky status. User is intentionally re-adding it, so acceptance is implicit. |
| Anthropic-format streaming differs from OpenAI-format and breaks clients | Low (already supported via existing `openAIToAnthropicPayload` and streaming paths used by Minimax-M3 testing earlier) | Existing tests cover Anthropic payload conversion. New test covers streaming end-to-end path for `proxy/qwen3.7-max`. |
| 1M context overflows message transform | Low | `truncateMessages` uses per-brain `context`; new entries declare 1M. |
| Pricing rounding differs in different sources | Low | Use OpenCode Go's published numbers (single source of truth). |
